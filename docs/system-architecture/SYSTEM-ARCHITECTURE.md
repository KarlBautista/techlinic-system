# System Architecture — TechClinic System

The system architecture diagram illustrates the three-tier layered architecture of the TechClinic: Health Record Management System. It follows the standard Presentation–Application–Data pattern, with clear separation of concerns between the user interface, business logic, and data persistence layers, ensuring modularity, maintainability, and security throughout the system.

The Presentation Layer is built using React 19 with Vite as the build tool, running on port 5173. This layer is responsible for rendering the user interface within the web browser and managing all client-side interactions. It comprises fourteen page components — including the Landing Page for student self-registration, the Dashboard for clinic overview, Patient Records for visit history, Add Diagnosis for clinical documentation, Medicine Inventory for stock management, Analytics for data visualization, and Notifications for alert management. Application state is managed through five Zustand stores (authentication, patient data, medicine, charts, and notifications) with local storage persistence, ensuring data consistency across user sessions. The layer utilizes React Router for client-side navigation, ApexCharts for interactive data visualizations (area charts, donut charts, Pareto charts, and bar charts), Tailwind CSS for responsive styling, Axios for HTTP communication with the backend, SweetAlert2 for user-friendly dialogs, Tanstack Table for data table management, and Lucide React for consistent iconography. A direct Supabase Auth Client is also integrated within this layer, enabling secure authentication operations (sign-in, sign-out, and session management) without routing through the backend.

The Application Layer is powered by Express.js 5 running on Node.js at port 3000. This layer serves as the intermediary between the frontend and the database, exposing a RESTful API that handles business logic, data validation, and authorization. It consists of seven route modules (data, medicine, analytics, diseases, notifications, authentication, and public routes) that define the API surface, five controller modules that implement business logic for patient records, medicine inventory, analytics aggregation, disease management, and notification delivery, and two middleware modules for JWT-based authentication and request validation. The backend connects to Supabase through two client configurations: the Anon Client, which uses the anonymous key and respects Row Level Security policies for standard data operations, and the Admin Client, which uses the service role key to bypass RLS for administrative operations such as creating and deactivating user accounts. Additional dependencies include CORS for cross-origin communication between the frontend and backend, dotenv for environment variable management, and Moment.js for date and time formatting.

The Data Layer is provided by Supabase, a Backend-as-a-Service platform built on PostgreSQL. It stores all persistent data across eight tables: records (patient visit entries), patients (unique student directory), diagnoses (clinical assessments linked to visits), medicines (inventory with stock tracking), diseases (master disease list for diagnosis and outbreak monitoring), users (authenticated profiles with role assignments), notifications (system-generated alerts), and audit_trail (activity logs for accountability and compliance). All tables are protected by Row Level Security (RLS) policies that enforce access control at the database level. Supabase Auth provides email and password-based authentication with JWT token generation, session management, and password reset flows, and is accessed directly by the frontend for authentication and by the backend for token verification.

Together, these three layers ensure that the TechClinic System operates with a clear separation of responsibilities. The Presentation Layer handles user interactions, the Application Layer processes business logic and enforces security, and the Data Layer provides reliable and secure data persistence. This architecture supports independent development and maintenance of each layer, improves testability, and enforces access control at multiple levels — through frontend route guards, backend middleware authentication, and database-level Row Level Security — ensuring the system is both secure and scalable for managing student health records effectively.
