# Data Flow Diagram (DFD Level 1) — TechClinic System

The data flow diagram (DFD) presents the detailed flow of information within the TechClinic: Health Record Management System. It decomposes Process 0 from the context diagram into nine sub-processes, illustrating how data moves between the main processes, external entities, and data stores throughout the system.

The system begins with Process 1 (Patient Registration), which handles self-registration by students at a public kiosk. Patient information is stored in the Patients data store (D1) and a visit record is created in the Records data store (D2), while a registration notification is simultaneously sent to Process 7 (Notification & Alerts). The Nurse interacts with Process 2 (Record Management) to update and manage patient visit records, which are persisted in D2 and forwarded to Process 3 (Diagnosis Processing) for clinical assessment. The Doctor uses Process 3 to input diagnosis data — including disease, treatment, and medication — which is stored in the Diagnoses data store (D3) and the Diseases data store (D5). Process 4 (Medicine Inventory) allows the Nurse to manage medicine stock levels; dispensation records are stored in the Medicines data store (D4), and when stock falls below a threshold, a low-stock alert is triggered to Process 7. Process 5 (Analytics & Reporting) aggregates data from D2 (Records), D4 (Medicines), and D6 (Users) to generate reports and interactive charts accessible to Nurses, Doctors, and Admins. Process 6 (Authentication & Access Control) manages login credentials and user sessions for all staff roles, storing account data in the Users data store (D6). Process 8 (Certificate Generation) produces printable medical certificates and prescriptions by retrieving patient and diagnosis data from the relevant data stores. Process 9 (Personnel Management) allows Admins to manage the clinic staff directory and links attending physicians to diagnosis records.

The Admin interacts with the system through Process 6 for authentication, Process 9 for personnel management, and the Users data store (D6) for user activation and deactivation. In return, the system delivers a dashboard overview, analytics data, and activity logs from Process 5. The Supabase external entity serves as the backend-as-a-service platform that receives persisted data and authentication requests from the system and returns query results and authentication tokens.

Together, these processes and data stores demonstrate how the system efficiently handles, stores, and processes health-related data to support effective clinical operations. The diagram reveals the interdependencies between processes — such as how the registration process feeds into both record management and notifications, or how the diagnosis process connects to certificate generation and analytics — ensuring that the design promotes accuracy, security, and accessibility in managing student health information.
